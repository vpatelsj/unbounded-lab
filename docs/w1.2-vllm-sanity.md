# W1.2 vLLM sanity check — Qwen3-30B-A3B-GPTQ-Int4 on `spark-2c24`

**Status:** partial. The deployment ([`inference/vllm-qwen-moe/`](../inference/vllm-qwen-moe/))
is shipped and serving; smoke-test endpoint replies are recorded below. The
throughput sweep is pending W1.6 (the benchmark harness). This page is the
home those numbers land in, per the W1.2 scope line in
[`plan.md`](../plan.md): *"measure actual sustainable batch × context × prompt
throughput, document in `docs/`."*

> **Plan deviation — model variant.** The plan calls for BF16. Deployed
> variant is `Qwen/Qwen3-30B-A3B-GPTQ-Int4` because (a) ~65 GiB of
> unified memory on `spark-2c24` is pinned in unreclaimable host page cache
> (DGX OS denies `drop_caches`) and (b) vLLM v0.11's prebuilt CUTLASS
> scaled-mm has no sm_121a kernel, so FP8 fails before profile_run
> finishes. GPTQ-Int4 dispatches to Marlin and works. Deviation chain is
> documented in [`storage-pain-journal.md`](../storage-pain-journal.md) and
> [`inference/vllm-qwen-moe/README.md`](../inference/vllm-qwen-moe/README.md).

The plan flags this deployment as **on the edge of OOM for KV cache at 32K**. The
purpose of this sanity check is to replace that hand-wave with measured numbers
**before** anything in Wave 2 builds on top of it. With GPTQ-Int4 weights
(~15.6 GiB GPU footprint) plus the unreclaimable host cache, the working KV
pool is 9.31 GiB / 101 712 tokens — 3.10x concurrency at the configured 32K
max-model-len.

## What gets measured

Three scalar dimensions, swept independently against the deployed config (single
GB10, GPTQ-Int4, `--max-model-len 32768`, `--gpu-memory-utilization 0.22`,
`--tensor-parallel-size 1`, `--enforce-eager`, `VLLM_USE_FLASHINFER_SAMPLER=0`):

1. **Sustainable concurrent requests (batch).** Smallest count at which p99 TTFT
   stops being flat — the saturation point, not the OOM point.
2. **Sustainable context length.** Largest prompt length the server accepts at
   batch=1 without KV-cache eviction or `--max-model-len` rejection. The plan's
   "edge of OOM at 32K" claim gets a number here.
3. **Sustainable prompt throughput.** Tokens/s in and tokens/s out, at the
   saturation point from (1).

All runs hit `vllm.lab-vllm-qwen-moe.svc.cluster.local:8000/v1/completions` from
an in-cluster pod, so the path under measurement excludes the Ollama-compat
sidecar (that has its own row).

## Methodology (pending W1.6)

W1.6 specifies the harness; W1.2 just consumes it. Until W1.6 lands, the
placeholder methodology is:

```sh
# Once W1.6 ships:
make w1.6-bench MODEL=Qwen/Qwen3-30B-A3B-GPTQ-Int4 \
  ENDPOINT=http://vllm.lab-vllm-qwen-moe.svc.cluster.local:8000/v1 \
  PROFILE=w1.2-sanity \
  OUT=docs/w1.2-vllm-sanity.results.json
```

Until then, the manual smoke test in
[`inference/vllm-qwen-moe/README.md`](../inference/vllm-qwen-moe/README.md#reaching-the-api)
is the only verification that the endpoint responds at all.
